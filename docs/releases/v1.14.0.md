# v1.14.0 — 2026-07-17

Distributed-trace analysis tools — the one capability a TypeScript MCP
prototype (`ssn-bzrk`) had that this server didn't.

## What shipped

- `trace_find_slow` — highest-duration root spans in the time window.
- `trace_find_errors` — spans whose status indicates an error.
- `trace_analyze` — full breakdown of one trace by `trace_id`: every span
  in time order, plus correlated log lines sharing the same `trace_id`.

Field names (`trace_id`, `span_id`, `parent_span_id`, `span_name`,
`duration`, `status_code`) follow this table's existing `<signal>_name`
convention (`metric_name` for metrics, `body`/`severity_text` for logs).

## Shipped unverified, then live-verified

The initial commit shipped these tools flagged `UNVERIFIED`: the Berserk
query gateway was unreachable from every vantage point available at the
time, so the field names were a best-effort guess by analogy, not a live
capture.

The outage turned out to be a real incident, not a network fluke — VM-C's
whole stack had been down for ~25h after the `nursery` container's Docker
memory limit (256m) was too tight for its actual working set, causing
repeated OOM-kills that ended in the stack being manually stopped. Fixed by
raising the limit to 512m (matching the query/postgres/minio tier) and
restarting the stack.

With the cluster back, all six field names were confirmed correct — this
cluster does ingest real span data, from Berserk's own self-instrumented
services (`service=query`, `service=gateway`, `service=ingest`). Two real
bugs surfaced and were fixed in the same pass:

1. **`duration` is a *dynamic*-typed column.** Berserk's KQL engine rejects
   `sort by duration` directly. `trace_find_slow` now casts with
   `toint(duration)` first.
2. **Not every row sharing a `trace_id` is a span.** Other correlated
   telemetry (e.g. a log row) can carry the same `trace_id`/`span_id` with
   a null `span_name`, which gave child-before-parent ordering when sorted
   by `timestamp`. `trace_analyze` now filters to `isnotnull(span_name)`
   and sorts by `start_time`.

Tool descriptions and the README were updated to drop the `UNVERIFIED`
framing once this was confirmed against live data.
