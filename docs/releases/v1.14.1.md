# v1.14.1 — 2026-07-18

Fixes a silent-failure bug in the three agent-analytics tools shipped in
v1.12.0, found while live-verifying `claude_token_burn` against real data.

## The bug

`_json_records()` only recognized a bare JSON array or a
`{"rows"/"data"/"results"/"records": [...]}` wrapper of row-dicts. Real
`bzrk --json` output is shaped
`{"Tables": [{"schema": {"columns": [...]}, "rows": [[...]]}], ...}` —
positional arrays matching column order, not dicts. That shape went
unrecognized, so parsing silently fell through to a jsonl/table parser that
found nothing: every call returned "No Claude Code events found" against
real data, with no error, just silence.

## The fix

`_json_records` now unwraps `Tables[0].rows` against
`Tables[0].schema.columns`. This also confirmed `claude.tokens_input` and
`claude.tokens_output` are the real ingest attribute names (previously an
unverified guess in a code comment).

Verified against a real 7-day query: 5 sessions, exact token counts, and a
genuine top token-burning session identified — 1,592 events, Opus-heavy,
73% of the week's total spend, confirmed not a loop via `claude_loop_check`
(just large individual turns).

Added test coverage for the real `Tables` shape, which the existing suite
never exercised (only jsonl and the imagined `{"rows": [...]}` wrapper) —
the exact gap that let this go undetected.

## Also in this release

Scrubbed a homelab-specific hostname ("VM-C") and its exact
docker-compose memory-limit setup, which had leaked into the v1.14.0
README writeup and a code comment. Kept the operationally useful content
(how to tell an OOM crash-loop apart from a network/auth issue) while
removing the operator-specific details.
