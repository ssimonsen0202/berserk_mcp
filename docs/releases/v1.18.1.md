# v1.18.1 — 2026-07-23

Fixes the `windows-latest` CI failure that started at v1.16.0 and persisted
through v1.17.0 and v1.18.0: `test_concurrent_persist_learned_query_loses_no_entries`
failing on Python 3.11/3.12 (Python 3.9 and every `ubuntu-latest` job stayed
green throughout).

## Root cause

```
FAIL: test_concurrent_persist_learned_query_loses_no_entries
AssertionError: Lists differ: [PermissionError(13, 'Permission denied')] != []
```

`_FileLock.__enter__` (in both `berserk_mcp.py` and its duplicate in
`parser_factory.py`) creates a lock file with
`os.open(path, O_CREAT | O_EXCL | O_WRONLY)` and only caught
`FileExistsError` on contention. Under 12 threads racing to acquire and
release the *same* lock file, Windows can raise `PermissionError` instead of
`FileExistsError` for this exact call: NTFS briefly keeps a just-deleted
file in a pending-delete state, so a fast recreate from another thread's
`__exit__` (or the stale-lock-break path) collides as "access denied"
rather than "already exists". This is a different call site than the
`os.replace` Windows flake fixed in commit `23b56c1a` — same underlying
"transient Windows filesystem state, not a real permissions problem"
category, but unprotected.

## Fix

Both `_FileLock.__enter__` implementations now catch `(FileExistsError,
PermissionError)` together and treat a `PermissionError` here identically
to lock contention — retry with backoff, break the lock if it's stale, and
still raise `TimeoutError` if the condition persists past
`_LOCK_TIMEOUT_SECONDS`, so a genuine permissions problem is never silently
swallowed.

Not reproducible on macOS/Linux (POSIX `rename`/`open` semantics don't have
this quirk); confirmed via the actual Windows CI failure log rather than
guessed. 402/402 tests still pass locally; the fix's own effect can only be
confirmed by the next `windows-latest` CI run.
