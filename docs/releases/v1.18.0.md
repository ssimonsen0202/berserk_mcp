# v1.18.0 — 2026-07-23

Phase 2 (fleet-tenancy controls) and Phase 3 (three new native-analytics
tools) of the Berserk-native optimization program, both live-verified
against the `homelab` cluster.

## Phase 2: fleet-friendly operation

When many MCP instances share one Berserk cluster, each instance now limits
the load it contributes:

- **Worker startup jitter** (`BERSERK_WORKER_JITTER_SECONDS`, default
  `7200`) — randomizes `--worker` start time to prevent synchronized cron
  bursts. Interactive calls are never jittered.
- **Interactive tool budget** (`BERSERK_MCP_TOOL_BUDGET_SECONDS`, default
  `10`, clamped to `BZRK_TIMEOUT`) — a per-query budget for interactive
  `tools/call`; a timeout response advises a narrower window instead of
  hanging for the full `BZRK_TIMEOUT`.
- **Timeout cooldown** (`BERSERK_MCP_FAIL_COOLDOWN_SECONDS`, default `30`) —
  suppresses identical timeout retries within one MCP process, absorbing
  retry storms without delaying a retry with changed arguments.
- **Read-only result cache** (`BERSERK_MCP_CACHE_TTL_SECONDS`, default
  `120`) — short-TTL cache for allowlisted read-only rollups. Cached
  results are marked `(cached, <age>s old)`; mutation, discovery,
  generation, and arbitrary-search tools are never cached.

### Where the defaults came from

Each default is a measured value from a deterministic offline simulation
(1,000+ trials each, machine-readable results under `evals/results/`), not
an arbitrary guess:

| Control | Selected value | Evidence |
|---|---:|---|
| Worker startup jitter | 7,200 s | `jitter_sim.json` — at 100 workers, the smallest tested value keeping P(peak > 3 in 10 seconds) below 5% (4.0%). |
| Interactive tool budget | 10 s | Five repeats across every read-only fixed-query tool: overall p95 1.75 s; `ceil(1.5 × p95)` floors to 10 s, with no tool over budget. |
| Read-only result cache | 120 s | 69.0% replay hit rate with p95 staleness 112.6 s; 300 s reaches 82.0% hit rate but increases staleness. |
| Identical-timeout cooldown | 30 s | Absorbs five repeated storm calls without delaying changed-argument retries. |

The first broad-window latency sweep was stopped after `claude_cost_report`
exceeded a practical interactive runtime; the evaluator now defaults to a
bounded 15-minute window. A five-repeat live sweep against `homelab`
completed with zero errors (overall p95 1.75s), giving the budget rule
`ceil(1.5 × overall_p95), floor 10s, cap BZRK_TIMEOUT`. The implementation
keeps the safer configured 60-second ceiling until an operator explicitly
adopts the tighter value. Raw results: `evals/results/latency_eval.json`.

## Phase 3: three new native-analytics tools

- **`detect_anomalies`** (`sre`, `soc`) — statistical anomaly detection for
  service event volume, using zero-filled `make-series` plus
  `series_decompose_anomalies`. Optional service filter.
- **`forecast_capacity`** (`sre`) — fits an allowlisted host gauge with
  native `series_fit_line`; reports R² and slope, and refuses weak or
  downward trends rather than inventing a ceiling date.
- **`find_similar`** (`sre`, `soc`) — meaning-based log search when
  Berserk semantic indexing is enabled; otherwise gives an exact-term
  `search` fallback rather than pretending to be semantic.

All three accept the standard optional `since` window. Inputs are validated
before query construction: service and host names use a restricted
identifier character set, forecast metrics are allowlisted, similarity
descriptions reject query delimiters and are capped at 500 characters, and
`k` is bounded to 1–50.

### Live probes against `homelab`

- `detect_anomalies` parsed and returned zero-filled event, anomaly, score,
  and baseline arrays.
- `forecast_capacity` parsed native fit arrays. It refused a weak/downward
  OpenClaw trend (R²=0.156, negative slope) and reported a reliable
  HermesRuntime upward trend (R²=0.688) without inventing a ceiling date.
- `find_similar` degraded safely: this cluster's parser rejects
  `similarto`, so the response directs callers to exact-term `search`
  instead of silently falling back to fuzzy matching.

The router-harness plumbing and mock baseline run, but a real model gate
was not exercised — neither `OPENAI_API_KEY` nor a local Ollama endpoint
was configured in this environment. The mock baseline
(`evals/results/mock_mock-20260723-113523.json`) is retained for
repeatability, not as a quality claim for a production model.
