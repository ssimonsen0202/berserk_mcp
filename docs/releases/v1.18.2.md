# v1.18.2 — 2026-07-23

Fixes a real usability regression from v1.18.0's interactive tool budget,
found immediately on first real use: a flat 10s budget applied to every
`tools/call` regardless of the requested time window.

## The problem

Asked for a 72-hour, per-model usage report. Every attempt — `search` with
`make-series`, `claude_cost_report` at 72h/24h/6h, even a plain `count()`
aggregate — failed with:

```
<tool> exceeded its 10s query budget for window '72h ago'.
Retry with a narrower 'since' window.
```

Bypassing the MCP layer and running the same query directly against `bzrk`
confirmed it: the real query took **~12.8s**. Not a runaway query — the
default `default` table is a shared, high-volume firehose where only
`timestamp` is indexed, so a 72h scan is inherently more expensive than a
15-minute one, regardless of query shape (a bare `count()` over 72h timed
out just as surely as a `make-series`). The 10s figure itself was correctly
measured — it came from a five-repeat live sweep — but that sweep used
short (15-minute) windows, and the resulting number then got applied
uniformly to every window size, including `claude_cost_report`'s own 7-day
default, which the tool could now never satisfy.

Narrowing the KQL with an explicit `where timestamp >= ago(X) and
timestamp < ago(Y)` bound didn't help either: this engine's cost scales
with the outer `since`-implied scan range, not just the final row count
after KQL-level filtering.

## The fix

The budget now scales with the requested window instead of applying the
short-window number to every call:

```
effective_budget = base_budget + BUDGET_PER_HOUR_SECONDS * window_hours
                    (capped at BZRK_TIMEOUT)
```

New env var `BERSERK_MCP_BUDGET_PER_HOUR_SECONDS` (default `0.5`). With the
defaults: a 15-minute window still gets ~10.1s (effectively unchanged), a
72-hour window gets ~46s, and a 7-day `claude_cost_report` gets ~94s.
Setting it to `0` restores the old flat-budget behavior exactly (locked by
its own test).

Three new tests: the scaling calculation itself, the `BZRK_TIMEOUT` cap
being honored even at an absurd per-hour rate, and the `0`-disables-scaling
case matching the prior flat behavior bit-for-bit.

405/405 tests pass (up from 402 at v1.18.1).
